# Поддержка и сопровождение

Документ описывает эксплуатационный сценарий, роли и инструктаж администратора.

## 1. Цель сопровождения

- Обеспечить стабильность работы FinTrack.
- Сократить время восстановления при инцидентах.
- Поддерживать актуальность конфигурации, миграций и безопасности.

## 2. Сценарий развертывания (эталон)

### 2.1 Подготовка

1. Проверить актуальность ветки релиза.
2. Выполнить локально:

```bash
npm ci
npm run typecheck
npm run lint
npm test
npm run build
```

3. Убедиться, что для окружения заданы обязательные переменные:

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`
- `VITE_SUBSCRIPTIONS_PROVIDER`

### 2.2 База данных

1. Применить миграции Supabase по timestamp.
2. Проверить таблицы:

- `notification_preferences`
- `subscription_state_overrides`
- `subscription_usage_marks`

3. Проверить RLS-доступ под `authenticated` пользователем.

### 2.3 Деплой приложения

Вариант 1 (Docker):

```bash
docker compose up --build -d
```

Вариант 2 (статическая сборка):

```bash
npm run build
npm run preview
```

### 2.4 Post-deploy smoke

1. Вход в систему.
2. Создание транзакции.
3. Создание бюджета и проверка уведомления.
4. Экспорт CSV/Excel/PDF.
5. Проверка раздела подписок.
6. Проверка импорта из банков (симуляция).

## 3. Инструктаж администратора

### 3.1 Ежедневный регламент

1. Проверить доступность приложения.
2. Просмотреть критические ошибки фронтенда и Supabase.
3. Проверить инциденты и статус восстановления.

### 3.2 Еженедельный регламент

1. Проверить health ключевых пользовательских сценариев.
2. Контролировать рост данных в `transactions`, `notifications`, `recommendations`.
3. Проверить неуспешные попытки авторизации.

### 3.3 Ежемесячный регламент

1. Проверить зависимости (`npm audit`).
2. Обновить пакеты безопасности.
3. Провести ревизию доступов по принципу least privilege.
4. Проверить корректность и восстановимость бэкапов.

## 4. Матрица ответственности

- Администратор платформы: инфраструктура, доступность, rollback.
- Ответственный за БД: миграции, целостность данных, RLS.
- Frontend-инженер: UX smoke, регрессии UI, экспорты.
- Incident owner: коммуникация, таймлайн, RCA.

## 5. Инциденты и восстановление

### 5.1 Severity

- `P1`: недоступен вход или критический бизнес-поток.
- `P2`: частичная деградация (например, отчеты без экспорта).
- `P3`: некритичный дефект интерфейса.

### 5.2 Шаги реакции

1. Зафиксировать симптом и время начала.
2. Оценить затронутые функции и пользователей.
3. Назначить owner.
4. Временно ограничить влияние (feature flag/rollback).
5. Восстановить сервис.
6. Подготовить RCA и preventive actions.

## 6. Резервное копирование

- Частота: ежедневно.
- Минимальная цель: RPO <= 24 часа, RTO <= 4 часа.
- Drill восстановления: 1 раз в квартал.

## 7. Администраторские команды

```bash
# Проверка локальной готовности
npm run typecheck && npm run lint && npm test

# Smoke E2E
npm run test:e2e:smoke:ci

# Локальный контейнерный запуск
docker compose up --build
```

## 8. Каналы поддержки

- Канал ошибок приложения: issue tracker проекта.
- Канал инцидентов production: выделенный чат/почта команды.
- Эскалация P1: немедленно ответственному за релиз и администратору БД.
